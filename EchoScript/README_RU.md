# plic — chatlang

Крошечный самодельный язык («chatlang», файлы `.cl`), а также древовидный интерпретатор и TCP-чат-сервер, написанный на нём, отправленный на langjam plic. Платформа: **CPython** (без внешних зависимостей — только стандартная библиотека).

## Запуск

```bash
# запустить сервер (по умолчанию examples/chat.cl на порту 9000)
python3 main.py chat.cl 9000

# в других терминалах подключайтесь с помощью встроенного клиента...
python3 client.py 127.0.0.1 9000

# ...или обычным TCP-клиентом вроде netcat / telnet
nc 127.0.0.1 9000
```

Вас попросят ввести имя пользователя, после чего вы сразу окажетесь в комнате. Введите строку, чтобы отправить её всем. Встроенные команды из `examples/chat.cl`: `/history`, `/users`, `/quit`.

## Что внутри

| Требование                                        | Где                                                          |
|----------------------------------------------------|----------------------------------------------------------------|
| Исходный код языка (лексер/парсер/интерпретатор)   | `lexer.py`, `chatlang_ast.py`, `parser.py`, `interpreter.py`  |
| Рантайм чата + встроенные функции                 | `runtime.py`                                                  |
| Чат-сервер (добавление/рассылка/история/удаление пользователя) | `server.py`                                       |
| Чат-программа, написанная *на* chatlang            | `examples/chat.cl`                                             |
| Тестовый клиент                                    | `client.py`                                                    |

## Язык: chatlang

chatlang — небольшой динамически типизированный, событийно-ориентированный скриптовый язык. Файл `.cl` представляет собой последовательность объявлений `var`/`func` верхнего уровня и обработчиков `on <event>(...)`; сервер вызывает эти обработчики по мере подключения и отключения соединений.

### Типы
`number` (int/float), `string`, `bool`, `null`, `list` (`[1, 2, 3]`),
`dict` (`{"a": 1, "b": 2}`).

### Переменные
```
var count = 0
count = count + 1
```

### Условия
```
if (count > 10) {
    print("занято")
} else if (count > 0) {
    print("есть люди")
} else {
    print("пусто")
}
```

### Циклы
В chatlang есть `while` (отдельный примитив рекурсии не нужен, так как пользовательские функции могут вызывать сами себя):
```
var i = 0
while (i < len(names)) {
    print(names[i])
    i = i + 1
}
```

### Функции
```
func greet(name) {
    return "привет, " + name
}
```

### Коллекции
```
var users = ["alice", "bob"]
push(users, "carol")

var scores = {"alice": 10}
set(scores, "bob", 5)
print(get(scores, "bob"))
```

### Обработчики событий (основная логика чата)
Сервер вызывает следующие три обработчика; каждый параметр передаётся самим сервером.
```
on connect(id, username) { ... }        // вызывается при подключении пользователя
on message(id, username, text) { ... }  // вызывается на каждую отправленную им строку
on disconnect(id, username) { ... }     // вызывается при отключении пользователя
```

### Встроенные функции
- **Рантайм чата**: `broadcast(text)`, `send(id, text)`, `history(n)`,
  `users()`, `disconnect(id)`
- **Общего назначения**: `len`, `push`, `pop`, `keys`, `values`, `get`,
  `set`, `str`, `num`, `upper`, `lower`, `join`, `split`, `now`, `print`

## Архитектурные заметки

- `lexer.py` → токены → `parser.py` → AST (`chatlang_ast.py`) →
  `interpreter.py` обходит дерево с использованием объектов `Environment` с лексической областью видимости (замыкания работают: функции захватывают область видимости, в которой они определены).
- `runtime.py` владеет общим состоянием чата (подключённые клиенты, история сообщений) и предоставляет его скриптам только через вышеописанные встроенные функции — сам скрипт никогда не работает с сокетами напрямую.
- `server.py` запускает один поток на каждое TCP-соединение (только цикл приёма/чтения). Все вызовы интерпретатора проходят через единую блокировку, поэтому выполнение обработчиков сериализовано — общие `var` в скрипте `.cl` и история сообщений ведут себя как единый почтовый ящик-актор, без необходимости настоящих STM/блокировок внутри самого языка.

## Известные упрощения (MVP-объём для первого дня)

- Нет пользовательской обработки ошибок (`try`/`catch`) — ошибки времени выполнения в обработчике перехватываются сервером и логируются, не приводя к падению других клиентов.
- Нет персистентности — история сбрасывается при перезапуске сервера.
- Нет `/kick`, комнат или авторизации — их несложно добавить в виде дополнительных встроенных функций в `runtime.py` плюс немного логики в `examples/chat.cl`.
